# Academic Differences API

Простое и мощное API на Django REST Framework для управления расхождениями в учебных планах (РУП).

Это **полностью автоматизированный шаблон проекта**. Он включает в себя все необходимое для разработки, тестирования и
развертывания на любом Linux-сервере.

## ✨ Возможности

- **REST API**: Полный набор CRUD-операций для студентов, кафедр, предметов и академических расхождений.
- **Аутентификация**: JWT (JSON Web Tokens) для безопасного доступа к API.
- **Инфраструктура как код**: Весь процесс развертывания автоматизирован с помощью Ansible и Docker.
- **Процесс разработки**: Проект следует стратегии **Trunk-Based Development** с обязательными проверками через Pull
  Request'ы.
- **CI/CD**: Настроен pipeline на GitHub Actions для автоматических проверок, сборки Docker-образа и ручного деплоя.
- **Стиль коммитов**: Внедрена проверка на соответствие
  стилю [Conventional Commits](https://www.conventionalcommits.org/).
- **Качество кода**: Настроен `pre-commit` с хуками для `black`, `isort` и `pylint`.
- **Тестирование**: Высокое покрытие кода тестами с использованием `pytest` и `factory-boy`.
- **Docker-ready**: Проект полностью готов к контейнеризации и развертыванию.

## 🚀 Процесс разработки

### 1. Локальная разработка

1. **Клонируйте репозиторий** и перейдите в него.
2. **Настройте окружение** (рекомендуется `docker-compose`):
    - Создайте файл `.env` (можно скопировать из `.env.example`).
    - Запустите: `docker-compose -f docker-compose.dev.yml up --build -d`
3. **Установите pre-commit хуки**: `pre-commit install --hook-type pre-commit --hook-type commit-msg`
4. **Создайте feature-ветку** (`git checkout -b feat/my-new-feature`).
5. **Пишите код и коммиты**, следуя [Conventional Commits](https://www.conventionalcommits.org/). Используйте
   `cz commit` для помощи.
6. **Отправьте ветку** (`git push`) и создайте Pull Request в `trunk`.

### 2. Управление секретами с помощью Ansible Vault

Проект использует **Ansible Vault** для безопасного хранения секретов приложения (пароли, ключи). Эти секреты **не
хранятся в открытом виде в Git**.

#### Основные файлы

- `infra/ansible/vars/vault.yml`: Зашифрованный файл с секретами. Он безопасен для хранения в репозитории.
- `infra/ansible/ansible.cfg`: Содержит настройки Ansible, включая путь к `vault.yml`.

#### Работа с Vault

Для редактирования зашифрованного файла используется команда `ansible-vault`.

1. **Редактирование секретов:**
   ```bash
   ansible-vault edit infra/ansible/vars/vault.yml
   ```
   Ansible запросит пароль от Vault. После ввода пароля откроется редактор, где вы можете изменить секреты.

2. **Создание нового зашифрованного файла:**
   Если вы создаете новый проект, сначала создайте файл:
   ```bash
   ansible-vault create infra/ansible/vars/vault.yml
   ```
   Затем введите пароль и добавьте свои секреты.

3. **Просмотр содержимого:**
   ```bash
   ansible-vault view infra/ansible/vars/vault.yml
   ```

**Важно:** Пароль от Vault **не должен** храниться в репозитории. Он должен быть известен только членам команды, которым
это необходимо. Для CI/CD пароль передается через GitHub Secrets (`ANSIBLE_VAULT_PASSWORD`).

### 3. Добавление нового секрета в репозиторий

Если вы добавляете новую переменную, которая содержит чувствительную информацию (например, `API_KEY`), выполните
следующие шаги:

1. **Добавьте переменную в `infra/ansible/vars/vault.yml`:**
   ```yaml
   # infra/ansible/vars/vault.yml
   SECRET_KEY: "..."
   # ... другие секреты
   NEW_API_KEY: "your-new-api-key-here" # <--- Добавьте новую переменную сюда
   ```
   (Используйте `ansible-vault edit` для этого).

2. **Используйте переменную в шаблонах Ansible:**
   В шаблонах (например, `templates/.env.academic-api.prod.j2`) используйте переменную так же, как и другие:
   ```
   # templates/.env.academic-api.prod.j2
   NEW_API_KEY={{ NEW_API_KEY }}
   ```

3. **Обновите `README.md` или `docs/` (опционально):**
   Добавьте описание новой переменной в документацию, чтобы другие разработчики знали, для чего она нужна.

4. **Закоммитьте изменения:**
   Зашифрованный файл `vault.yml` можно безопасно коммитить в Git.

5. **Обновите GitHub Secrets (для CI/CD):**
   Убедитесь, что пароль от Vault (`ANSIBLE_VAULT_PASSWORD`) установлен в `Settings -> Secrets and variables -> Actions`
   вашего репозитория на GitHub.

### 4. CI/CD Процесс

- **На каждый Pull Request**: Запускается `ci.yml` workflow, который выполняет линтинг коммитов, линтинг кода и
  прогоняет тесты на нескольких версиях Python.
- **При слиянии в `trunk`**: После успешного прохождения всех проверок и слияния PR, `ci.yml` дополнительно собирает и
  архивирует Docker-образ как артефакт сборки.

### 5. Развертывание (Deployment)

Развертывание производится **только** через GitHub Actions. Подробная инструкция по настройке и запуску деплоя находится
в `docs/DEPLOYMENT.md`.

## 🛠 Инструменты для разработки

#### Тестирование

- **Запуск**: `pytest`
- **Запуск с отчетом о покрытии**: `pytest --cov` (использует настройки из `.coveragerc`)

#### Pre-commit хуки

Хуки (`black`, `isort`, `pylint`, `commitizen`) устанавливаются командой
`pre-commit install --hook-type pre-commit --hook-type commit-msg` и запускаются автоматически перед каждым коммитом.
